2.5 프롬프트 템플릿 활용법

자주 사용하는 프롬프트를 템플릿화하면 일관된 품질의 문서를 빠르게 생성할 수 있습니다.

실무자용 프롬프트 템플릿 라이브러리

문서 유형 핵심 프롬프트 구조 사용 시점
PRD 역할 + 배경 + 기능 개요 + PRD 구조 + 형식 신규 기능 기획 시
사용자 스토리 페르소나 + 기능 설명 + "As a... I want... So that..." 형식 스프린트 계획 시
경쟁 분석 자사 정보 + 비교 항목 + 테이블 형식 + SWOT 전략 수립 시
릴리스 노트 변경사항 목록 + 대상 독자(사용자) + 톤앤매너 배포 직전
주간 보고서 이번 주 성과 + 다음 주 계획 + 이슈/블로커 + 형식 매주 금요일
팁: 프롬프트 템플릿을 노션이나 메모앱에 저장해두고, 상황에 맞게 [변수] 부분만 교체하여 사용하면 효율적입니다.

실습 과제

위 5가지 문서 유형 중 2개를 선택하여 여러분만의 프롬프트 템플릿을 만들어보세요. 실제 프로젝트 맥락을 반영하면 더욱 좋습니다.

2.6 PL을 위한 프로젝트 관리 문서

PL 관점: PM이 "무엇을 만들 것인가"를 정의하는 문서를 작성한다면, PL은 "어떻게, 언제까지 만들 것인가"를 관리하는 문서를 작성합니다.

실무 문서 체계 관계도

graph LR subgraph PM 영역 A[PRD
무엇을 만들 것인가] --> B[기획서/제안서] A --> C[요구사항 정의서] A --> D[스펙 문서/API 명세] end subgraph PL 영역 E[WBS
작업 분해 구조] --> F[프로젝트 일정표] E --> G[리스크 관리대장] F --> H[주간 보고서] end A -->|요구사항 전달| E C -->|범위 확정| F

PL 핵심 문서 유형

문서 목적 AI 활용 방법
WBS (작업 분해 구조) 프로젝트를 관리 가능한 작업 단위로 분해 기능 목록 입력 → AI가 하위 작업 분해
프로젝트 일정표 마일스톤과 작업별 일정 관리 WBS + 리소스 → AI가 간트차트 구조 생성
리스크 관리대장 프로젝트 리스크 식별 및 대응 계획 프로젝트 맥락 → AI가 리스크 도출 및 대응방안 제시
주간 상태 보고서 진척 현황, 이슈, 다음 주 계획 공유 메모/데이터 입력 → AI가 구조화된 보고서 작성
이슈 로그 발생한 이슈 추적 및 해결 현황 관리 이슈 목록 → AI가 영향도 분석 및 우선순위 제안
WBS 생성 프롬프트
당신은 10년 경력의 프로젝트 리더(PL)입니다.

다음 프로젝트의 WBS(Work Breakdown Structure)를 작성해주세요.

## 프로젝트 개요
- 프로젝트명: 고객 피드백 자동 분류 시스템
- 기간: 12주
- 팀 규모: 개발 3명, QA 1명, 디자이너 1명

## WBS 작성 조건
1. 3단계 깊이로 분해 (Phase → Task → Sub-task)
2. 각 작업에 예상 소요일 포함
3. 선행 작업(dependency) 관계 표시
4. 마일스톤 3~4개 지정
5. 테이블 형식으로 정리:
   | WBS ID | Phase | Task | Sub-task | 담당 | 소요일 | 선행작업 |
리스크 관리대장 프롬프트
다음 프로젝트의 리스크 관리대장을 작성해주세요.

## 프로젝트 맥락
- AI 기반 고객 피드백 분류 시스템 (12주)
- 외부 AI API 연동 필수
- 개인정보 포함 데이터 처리

## 리스크 관리대장 형식
| ID | 리스크 | 카테고리 | 발생확률 | 영향도 | 위험등급 | 대응전략 | 트리거 | 담당자 |

- 기술, 일정, 리소스, 외부 카테고리 포함
- 최소 10개 리스크 식별
- 대응전략: 회피/전이/완화/수용 중 선택
- 트리거: 리스크가 현실화되는 조건
주간 상태 보고서 프롬프트
다음 정보로 PL 주간 상태 보고서를 작성해주세요.

[이번 주 메모]
(작업 현황, 이슈, 특이사항을 붙여넣기)

보고서 형식:
1. 전체 진척률 (%) 및 상태 (정상/주의/위험)
2. 이번 주 완료 항목 (bullet point)
3. 진행 중 항목 및 잔여 작업
4. 이슈 & 블로커:
   | 번호 | 이슈 | 영향도 | 상태 | 대응방안 |
5. 다음 주 계획 (우선순위순)
6. 리스크 업데이트
7. 의사결정 필요 사항

실습 과제

현재 진행 중이거나 계획 중인 프로젝트를 하나 선택하여, AI로 WBS와 리스크 관리대장을 생성해보세요. AI가 도출한 결과를 검토하고 빠진 항목을 추가하는 것이 핵심입니다.

2.7 AI 문서 검토 및 품질 관리

핵심: AI가 생성한 문서를 그대로 사용하면 품질 문제가 발생합니다. PM/PL은 AI 결과물의 "편집자"이자 "품질 관리자" 역할을 해야 합니다.

AI 문서 품질 관리 프로세스

graph TD A[AI 문서 생성] --> B[정확성 검증
수치/날짜/고유명사] B --> C[일관성 확인
용어/표기법/어투] C --> D[맥락 적합성
조직 상황 반영 여부] D --> E[톤앤매너 조정
대상에 맞는 어투] E --> F{품질 기준 통과?} F -->|미달| G[AI에 수정 요청
구체적 피드백] G --> B F -->|통과| H[버전 관리 등록] H --> I[최종 승인 및 배포]

AI 문서 검토 체크리스트

검토 항목 확인 내용 흔한 AI 오류
정확성 수치, 날짜, 고유명사가 정확한가? 존재하지 않는 통계 인용, 날짜 오류
일관성 용어, 표기법, 어투가 문서 전체에서 통일되었는가? 같은 개념을 다른 용어로 표현
맥락 적합성 우리 조직/프로젝트 상황에 맞는 내용인가? 일반적인 내용만 나열, 구체성 부족
톤앤매너 문서 대상(경영진/개발팀/고객)에 맞는 어투인가? AI 특유의 과도하게 정중한 표현
완전성 빠진 섹션이나 누락된 요구사항이 없는가? 중요한 엣지 케이스 누락
실행 가능성 제안된 액션 아이템이 실제로 실행 가능한가? 비현실적인 일정이나 목표 제시

AI에게 AI 결과물을 검토시키기 (셀프 리뷰)

동일 AI에게 자신의 결과물을 검토하게 하거나, 다른 AI 도구로 교차 검토하면 품질을 높일 수 있습니다.

AI 셀프 리뷰 프롬프트
다음 PRD 문서를 시니어 PM의 관점에서 비판적으로 검토해주세요.

[검토 대상 문서]
(AI가 생성한 PRD를 붙여넣기)

검토 기준:
1. 누락된 요구사항이 있는가? (특히 엣지 케이스, 에러 처리)
2. 성공 지표(KPI)가 측정 가능하고 현실적인가?
3. 기술적으로 모호한 표현이 있는가?
4. 개발팀이 이 문서만으로 구현을 시작할 수 있는가?
5. 우선순위 기준이 명확한가?

발견된 문제를 심각도(상/중/하)와 함께 테이블로 정리하고,
각 문제에 대한 수정 제안을 제시해주세요.

버전 관리 워크플로우

  1. v0.1 — AI 초안

    AI가 생성한 원본을 저장합니다. 파일명에 "_AI초안"을 붙여 구분합니다.

  2. v0.2 — PM/PL 1차 수정

  3. v0.3 — AI 셀프 리뷰 반영

    AI에게 수정본의 재검토를 요청하고, 타당한 피드백을 반영합니다.

  4. v1.0 — 최종 확정

    이해관계자 리뷰를 거쳐 최종 버전을 확정합니다.

실습 과제

이전 섹션(2.1~2.6)에서 AI로 생성한 문서 중 하나를 선택하여, 위 체크리스트 6개 항목으로 직접 검토해보세요. AI 셀프 리뷰 프롬프트도 적용하여 개선점을 비교해보세요.

2.8 스펙 문서 및 API 명세서 작성

PM이 작성할 수 있는 기술 문서 유형

문서 유형 목적 AI 활용 방법
기능 명세서 기능의 상세 동작 방식을 정의 사용자 스토리 → AI가 상세 명세로 확장

데이터 사전 시스템에서 사용하는 데이터 항목 정의 기능 목록 → AI가 필요한 데이터 필드 도출

## 기능 설명
- 기능: 팀 멤버 초대 및 권한 관리
- 사용자가 할 수 있는 것:
  1. 이메일로 팀 멤버 초대
  2. 멤버의 권한 변경 (관리자/편집자/뷰어)
  3. 멤버 제거
  4. 초대 현황 조회 (대기중/수락/거절)

## 작성 요청
각 기능별로 다음 형식으로 정리해주세요:

### API 엔드포인트
- URL: /api/...
- Method: GET/POST/PUT/DELETE
- 설명: 이 API가 하는 일

### 요청 데이터
| 필드명 | 타입 | 필수 여부 | 설명 |

### 응답 데이터
| 필드명 | 타입 | 설명 |

### 에러 케이스
| 상황 | 에러 코드 | 메시지 |

[참고] 이 문서는 개발팀이 상세 설계 시 참고할 초안입니다.
PM과 개발팀이 함께 리뷰하며 확정합니다.
화면 설계 명세서 프롬프트
다음 화면의 설계 명세서를 작성해주세요.

## 화면 정보
- 화면명: 팀 대시보드
- 목적: 팀장이 팀원들의 작업 현황을 한눈에 파악
- 접근 권한: 관리자, 팀장

## 명세서 포함 항목
1. 화면 구성 요소 목록
   | 영역 | 구성 요소 | 데이터 | 동작 |
2. 사용자 인터랙션 시나리오
   - 클릭, 필터, 정렬 등 각 동작의 결과
3. 상태별 화면 변화
   - 데이터 없음, 로딩 중, 에러 상태
4. 반응형 대응
   - 데스크톱/태블릿/모바일별 레이아웃 차이
5. 접근성 고려사항
중요: AI가 생성한 기술 명세서는 반드시 개발팀과 함께 리뷰해야 합니다. PM의 역할은 "무엇이 필요한지"를 명확히 전달하는 것이고, "어떻게 구현할지"는 개발팀이 결정합니다.

실습 과제

여러분의 제품에서 하나의 기능을 선택하여 AI로 API 명세서 초안을 생성해보세요. 개발팀이 이해할 수 있는 수준인지, 빠진 엣지 케이스는 없는지 검토해보세요.
